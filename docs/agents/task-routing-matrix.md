# Task Routing Matrix / 任务派工决策表

> 目标：在接单前先判断“这类任务最适合走 safe-refactor-loop / CLI 自动化引擎，还是直接由有鱼在当前聊天窗口处理”。
> 作用：降低北冥的派工成本，避免把轻任务做重、把重任务做轻。
>
> 先记住一条总原则：
> - 任务越偏“多文件、多阶段、高风险、要留战时文档、要走审计/收编/自愈循环”，越应走 CLI 自动化引擎。
> - 任务越偏“单轮判断、小范围改动、语义靠人工把关、读完即可直接回读验收”，越适合当前聊天窗口直接处理。

## 1. 先问三个分流问题

### Q1：这次是否需要固定状态机与战时文档？
满足任一条，优先走 CLI 自动化引擎：
- 需要 Task Contract / Status Ledger / Verification Chain
- 需要多轮推进、自愈循环、阶段闸门
- 需要 M3 / 调用链 / pytest / 实现报告这一套固定复审链
- 需要主线收编级检查

### Q2：这次是否主要依赖“语义判断”而不是“流程编排”？
满足任一条，优先走当前聊天窗口直接处理：
- 任务核心是文案、术语、结构、法律语义是否准确
- 修改范围小，读完文件后可直接给出精修结果
- 验收主要靠回读、对照、逐条解释，而不是长链路自动化

### Q3：当前仓库是否干净，候选 diff 是否容易被污染？
满足任一条，需要先隔离 worktree：
- 当前主仓已有脏改动
- 任务会新增多份文档、现场文件或多轮迭代证据
- 任务需要形成可归因候选 diff
- 任务后续可能升级为主线收编、法典升级或代码战役

## 2. 分类决策表

| 任务类型 | 典型信号 | 建议执行方式 | 是否需要先隔离 worktree | 说明 |
|---|---|---|---|---|
| 纯文档精修 | 单文件或少量文件；核心是措辞、结构、路径、格式、术语顺直；不改执行逻辑 | 当前聊天窗口直接处理 | 默认否；若主仓已脏或会连带多文档联动，则改为是 | 这类任务胜负手在“人工语义把关 + 即时回读”，没必要把轻任务强行升格成完整自动化战役 |
| 法典升级 | 触碰宪章、协议、模板、状态机、审计规则、执行模式；需要解释“迁移/新增/删除” | 当前聊天窗口直接处理 | 是 | 法典升级更依赖有鱼与北冥的近距离语义对齐；即使在当前窗口执行，也必须按法典升级模式留合同、台账、验证链，并在隔离 worktree 内操作 |
| 主线收编 | 目标是把候选战果刻入 main；要做文件范围审查、代码/文档一致性、测试复验、Git 边界审查 | CLI 自动化引擎 | 是 | 这是固定法则最重、步骤最硬的一类任务；最适合交给 safe-refactor-loop 驱动固定检查，防止靠口头判断收编 |
| 技术债治理 | TDB-*、多文件收敛、需要小步推进、要防扩战线、要留证据 | CLI 自动化引擎 | 是 | 技术债治理天然适合“合同锁边界 + 审计 + 自愈循环”的自动化模式，能降低跑偏概率 |
| 高风险代码战 | 涉及安全边界、TTY、删除、PATH、profile、审批闸门、真实环境或契约级行为 | CLI 自动化引擎 | 是 | 这类任务绝不能只靠当前窗口的口头推进，必须让审计链、验证链和人工闸门一起工作 |
| 小范围即改即验任务 | 单点修补；影响面小；一轮修改后能立刻回读或跑最小验证；不依赖长链路协作 | 当前聊天窗口直接处理 | 默认否；若当前主仓已脏、或改动触及高风险文件，则改为是 | 关键在于“改得小、验得快、边界清楚”；这类任务应保持轻量，不要平白抬高派工成本 |

## 3. 一句话派工规则

- 如果任务目标是“把复杂流程管住”，派给 CLI 自动化引擎。
- 如果任务目标是“把语义与文案做准”，优先在当前聊天窗口处理。
- 如果当前主仓不干净，或任务需要形成独立候选 diff，先切 worktree，再谈由谁执行。

## 4. 最适合 CLI 自动化引擎的案例

### 案例 A：TDB-3 `uninstall` CLI 参数语义与实现对齐
- 为什么适合 CLI：
  - 属于技术债治理 + 高风险代码战叠加场景
  - 需要合同锁边界，防止顺手放松 typed confirmation 或 TTY 边界
  - 需要固定复审顺序：M3 审计 → 调用链差异 → pytest → 实现报告
  - 需要自愈循环，而不是一次改完就口头宣布完成
- 推荐执行方式：CLI 自动化引擎
- worktree：必须先隔离
- 参考依据：`docs/exec-plans/tech-debt-tracker.md`、`docs/agents/law-casebook.md` Case 6 / 7 / 8 / 9

### 案例 B：主线收编终审
- 为什么适合 CLI：
  - 需要严格执行主线收编协议的固定检查项
  - 需要逐文件审查、测试复验、Git 边界审查
  - 不应依赖临场口头判断“差不多可以推了”
- 推荐执行方式：CLI 自动化引擎
- worktree：必须先隔离
- 参考依据：`docs/agents/mainline-integration-protocol.md`、`docs/agents/law-casebook.md` Case 17 / 18 / 19

## 5. 更适合当前聊天窗口直接做的案例

### 案例 C：北冥汇报模板路径表达小范围精修
- 为什么更适合当前窗口：
  - 核心是模板文案、字段口径、路径表达是否合规
  - 需要有鱼在当前对话里直接对照法典语义做判断
  - 改完后通过回读与定向搜索即可快速验收
- 推荐执行方式：当前聊天窗口直接处理
- worktree：若主仓已脏或要留下独立候选 diff，则仍建议先切；若仓库干净且只是单文件小改，可不切

### 案例 D：单页法典文档措辞顺直
- 例子：只修 `docs/agents/beiming-constitution.md` 某一节的术语一致性、排版、表述硬度
- 为什么更适合当前窗口：
  - 任务核心是语义质量，不是流程编排
  - 范围足够小，回读即可直接验收
  - 若硬套完整自动化引擎，派工成本高于任务本身
- 推荐执行方式：当前聊天窗口直接处理
- worktree：默认不必；若当前主仓已脏，则先切

## 6. 默认落地口径

### 应优先派给 CLI 自动化引擎
- 主线收编
- 技术债治理
- 高风险代码战

### 应优先在当前聊天窗口直接处理
- 纯文档精修
- 法典升级
- 小范围即改即验任务

### 应默认先隔离 worktree 的情形
- 主线收编
- 技术债治理
- 高风险代码战
- 法典升级
- 当前主仓已脏的任何任务

## 7. 给北冥的最短派工口令

可直接按下面句式下令：

- 走 CLI：
  - “加载 safe-refactor-loop，接管【任务名】；先切独立 worktree，再恢复合同/台账/验证链，按 CLI 自动化引擎推进。”
- 走当前窗口：
  - “当前窗口直接处理【任务名】；不要升格成 CLI 自动化战役；如主仓已脏，先切 worktree 再改。”

## 8. 本表的边界

- 这是派工决策表，不是执行流程全文。
- 具体进入 CLI 自动化引擎后，仍服从 `~/.hermes/skills/safe-refactor-loop/SKILL.md`。
- 具体涉及主线收编时，仍服从 `docs/agents/mainline-integration-protocol.md`。
- 具体涉及法典升级时，仍需按法典升级模式交代迁移、删除、新增三类来源去向。
